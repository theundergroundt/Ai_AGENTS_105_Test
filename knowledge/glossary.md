# Glossary

## upstream

- 한 줄 정의: 내가 작업하는 저장소가 따라가는 원래 기준 저장소 또는 그 원격을 뜻한다.
- Git 맥락: 내 fork가 있을 때 원본 저장소를 `upstream` remote로 추가해 두는 경우가 많다.
- 왜 중요하나: 원본 저장소의 최신 변경을 내 fork나 로컬 작업본으로 가져올 때 기준이 된다.
- 예시:
  내가 `myname/Claw3D`를 fork했고 원본이 `iamlukethedev/Claw3D`라면,
  보통 내 fork는 `origin`, 원본은 `upstream`으로 둔다.

## fork

- 원본 저장소를 내 계정으로 복제한 별도 저장소다.
- 보통 원본 변경을 따라가면서 내 변경을 쌓기 위해 쓴다.

## origin

- 내가 직접 clone한 원격 저장소를 가리키는 기본 이름이다.
- fork 기반 작업에서는 대체로 내 fork가 `origin`이 된다.

## sandbox

- 한 줄 정의: 프로그램을 제한된 권한과 자원 안에서만 실행하게 하는 격리된 실행 공간이다.
- 왜 중요하나: agent가 로컬 파일, 네트워크, 시스템 명령에 무제한 접근하지 못하게 해서 사고 범위를 줄인다.
- 예시:
  특정 폴더만 읽고 쓰게 하거나, 허용된 사이트에만 접속하게 하는 실행 환경

## blueprint

- 한 줄 정의: 실행 환경을 어떤 설정과 정책으로 만들지 적어둔 배포 설계도다.
- 왜 중요하나: sandbox, 네트워크 정책, provider 연결 같은 구성을 재현 가능하게 만든다.
- 예시:
  어떤 컨테이너 이미지를 쓰고 어떤 정책을 적용할지 선언한 설정 파일 묶음

## runtime

- 한 줄 정의: agent나 프로그램이 실제로 실행되는 환경과 실행 계층을 뜻한다.
- 왜 중요하나: 같은 agent라도 어떤 runtime 위에서 도느냐에 따라 권한, 속도, 연결 방식이 달라진다.
- 예시:
  로컬 Python 프로세스, Docker 컨테이너, 원격 서버, sandbox 내부 실행 환경

## orchestrator

- 한 줄 정의: 여러 작업, agent, 상태, 이벤트의 흐름을 조정하는 제어 계층이다.
- 왜 중요하나: 어떤 요청을 누구에게 보내고, 언제 실행하고, 상태를 어떻게 업데이트할지 결정한다.
- 예시:
  assistant가 사용자 요청을 보고 Scheduler Agent나 Wellness Agent 중 하나를 선택하는 로직

## gateway

- 한 줄 정의: 외부 입력과 내부 시스템 사이를 연결하고, 요청·상태·이벤트를 중간에서 관리하는 진입 계층이다.
- 왜 중요하나: 여러 채널이나 클라이언트가 들어와도 내부 실행 흐름을 한곳에서 통제할 수 있다.
- 예시:
  웹 요청, 메시지 채널, device event를 받아 session과 state를 관리한 뒤 agent 실행으로 넘기는 계층
